# Diagram Rules

When explaining code flow, architecture, system design, service interactions, or data pipelines, include a Mermaid diagram. Use the appropriate diagram type:

- **Sequence diagrams** for request/response flows, API call chains, message passing between services
- **Flowcharts** for control flow, decision logic, processing pipelines
- **Class diagrams** for domain models, object relationships
- **ER diagrams** for database schemas, data models
- **State diagrams** for state machines, lifecycle transitions
- **C4 diagrams** for high-level system architecture

Always render diagrams in fenced ```mermaid blocks. Keep diagrams focused — one concept per diagram. If a flow is complex, break it into multiple diagrams rather than one sprawling chart.

Do NOT generate diagrams for trivial questions, single-file changes, or when the user just wants a quick answer.
